## 10.3 Extended protocols

Many disk drives also support common extended protocols that allows flexible access to higher-capacity disk formats and high-speed operation.

## Double density

Drives supporting true double density operation - 256 bytes per sector - also extend the read/write/put sector commands to use that size frame when accessing double density disks. Accesses to boot sectors (sectors 1-3) are an exception as they still use 128 byte data frames. The format command is also extended to return a 256 byte buffer when formatting a double-density disk.

## Double density boot sector encoding

Most disk drives that support double density write boot sectors as full 256 byte sectors, even though only 128 bytes are accessible to the computer. The first 128 bytes of the sector are used, with the last 128 bytes are ignored on read. The contents of the last 128 bytes are unspecified on write and vary between drives. The XF551 mirrors the first 128 bytes into the second 128 bytes, while the US Doubler will write stale data from the second half of its sector buffer into the second 128 bytes.

## High speed operation (US Doubler)

1050 disk drives with the ICD US Doubler modification can transfer data at higher speeds than regular 19,200 baud when sent special high-speed commands. This protocol was then also implemented by many other thirdparty disk drives.

The Return High Speed Index ($3F) command is used to query the high-speed index for a drive that follows the US Doubler protocol for high-speed operation. It is documented along with other US Doubler specific commands in Appendix G of the SpartaDOS Construction Set manual. It is a read command that ignores AUX1/2 and returns a data frame with a single byte indicating the POKEY divisor to use when communicating with the drive at high speed. This value is written into AUDF3, with AUDF4 set to $00. Standard 19,200 baud operation uses a divisor of $28 (18866-19040 baud), while the US Doubler uses a divisor of $0A (52160-52640 baud). ## Warning

The Return High Speed Index command overlaps with a Type 1 Poll (see 9.2 Poll Commands), which uses command $3F on an arbitrary device ID. One of the devices that answers a Type 1 Poll is the 850 Interface Module, which can respond to command $3F even for a disk drive device ID. In most cases a disk drive will answer command $3F on its device ID before the 850 does, but it is possible for both devices to conflict on the SIO bus, or for the 850 to answer on behalf of a non-existent disk drive.

There are a couple of ways to mitigate this conflict: avoid AUX1=$01, and either only send $3F after confirming that the disk drive exists and can ACK/NAK the command, or use AUX1 > $01 to disable the 850's handling of the command.

When a drive supports the US Doubler protocol, it accepts command frames at both normal 19200 baud and also the faster rate. The speed at which the command frame is sent determines the speed for the entire command, including ACK/NAK, complete/error, and data frames in both directions. No change to the command byte is necessary, nor are any high-speed flags used in the AUX bytes.

Command retry logic is essential when using the US Doubler protocol because many drives supporting it can only receive at one speed at a time and handle both speeds by toggling back and forth between them. This means that it is highly likely that the first command frame sent will be dropped. The TOMS Turbo drive is unusual in using the SIO clock lines to receive command frames at varying rates without alternating receive rates and relying on the computer to retry.

Not all disk drives use this method of high-speed communication. The Happy 810/1050, XF551, and Indus GT use differing methods that have no generic way of querying the drive, different high-speed command IDs, and use a low-speed command frame with a high-speed data frame. The 1050 Turbo uses bit 7 of AUX2 to enable high-speed data frame operation. Table 45 lists some known transfer rates used by disk drives.

## Percom block standard

High-capacity disk formats are almost universally managed by means of the Percom option block standard, established by Percom Data Corporation with their line of disk drives. The current disk format setting on the drives is described in a 12 byte block commonly called the Percom block, with a pair of query and set commands called Read Percom Block ($4E) and Write Percom Block ($4F). These were originally called the Option Table and the Read Option / Write Option commands in the original Percom RFD memo. 63

Note that all two-byte quantities in the Percom block are stored as big-endian (MSB first), because it was introduced on a 6809-based drive. This is opposite from the little-endian convention used by the 6502.

Most drives that support the Percom block only support the values 0 and 4 for the density byte. The Percom RFD-40S1 and the ATR8000 are exceptions as they also interpret bit 1 to enable clock-doubling for 8" disk operation. The Percom RFD directly ORs this value into the byte written into a hardware register, so all other bits must be 0 for normal operation.

Byte offset 9 contains a byte originally labeled by Percom as ACIA (not currently implemented). It is set to the ACIA configuration byte of $41 by the Percom RFD drive, but not actually used by the drive. No known drive interprets the value of this byte.

The Percom block is used both to query the format detected for a disk and to set the format to use when formatting a new disk. There is typically only one setting on the drive and setting it differently from the currently mounted disk can render the disk unreadable by the drive. The drive may also overwrite the setting if the disk is changed, a read/write operation is triggered that causes the drive to auto-detect the format on the current disk, or the user selects a density via external controls on the drive.

Drive firmware varies widely in handling of the Percom block. The step rate setting is frequently ignored and firmware usually only validates and stores just enough information for the formats supported by the drives. The XF551, for instance, barely checks enough information to be able to distinguish the four formats that it supports, and on a read request re-formats the Percom block based on that information. Other drives may use more of the block - particularly the original Percom drives - and so it is a good idea to read the Percom block from the drive and modify it rather than simply writing a new block to the drive. This also means that re-reading the block after a modification attempt is necessary to verify that the changes have actually been accepted by the drive. Drives do not necessarily fail an attempt to write an unsupported configuration block but may do so. The 810 Turbo, for instance, will return Error for a double-sided format but silently ignores other fields, and the Percom AT-88 will return Error on an attempt to select double density on a controller only capable of single density.

For formatting, the current Percom block setting is used for the next format command, usually $21 (format), $66 (format skewed), or $A1 (format with high-speed skew).

Drives may or may not update the Percom block state when detecting the density of a new disk. The US Doubler and Super Archiver, in particular, do not. ## Density reporting

After the disk drive detects the disk geometry, the computer also needs to know the detected geometry in some cases. An enhanced density disk can just be treated as a single density disk with more sectors since the sector size is the same, but for double density, the sector size must be known before reading sectors 4 or above.

Drives also vary in how the density is reported. The most common way is through bits 5 and 7 of the first drive status byte, with bit 7 having been introduced by the 1050 for enhanced density, and bit 5 for double density by Percom drives. Most, but not all drives, can also read back the detected disk geometry by reading back the Percom block. No known drives are able to detect whether the disk is double sided or has a non-standard geometry like non-standard track count or sectors per track.

## Double-sided format

A few disk drives are capable of reading and writing to both sides of the disk. This includes the XF551, ATR8000, and Percom drives. All disk drives that can do this support double density and manage enablement of the second side through the Percom block commands.

Of these disk drives, only the XF551 supports a status bit to indicate whether double sided move is selected (bit 6), and none of them can detect if a disk is double sided. The XF551 simply assumes and detects any double density disk as double sided.

## Double-sided sector mapping

The second side of the disk is exposed by an extension of the logical to physical sector mapping. Twice as many logical sectors are available, with the second half accessing the second side. For instance, a double-sided, double-density (DSDD) diskette has logical sectors 1-720 on the common side 0 and 721-1440 on side 1.

The precise mapping on the second side, however, differs between disk drives. There are four known mappings in use:

- The XF551 maps both tracks and sectors on the second side backwards. Logical sectors 721-738 map to physical sectors 18, 17, 16… 1 on side 1, track 39.
- The 1450XLD parallel disk drive maps tracks backwards but sectors forward on the second side. Logical sectors 721-738 map to physical sectors 1, 2, 3… 18 on side 1, track 39.
- Percom drives map the second side backwards similarly to the XF551, but have an error in the firmware that maps the second side incorrectly by one logical sector. Logical sectors 721-737 map to physical sectors 17, 16, 15… 1 on side 1, track 39 and logical sector 737 maps to side 1, track 38, sector 18. This means that the last logical sector 1440 is inaccessible as it maps to side 1, track 0, sector 0, which triggers a Record Not Found error when accessed. This bug is present in all known Percom RFD and AT firmware revisions.
- ATR8000 drives map the second side forwards in track and sector. Logical sectors 721-738 map to physical sectors 1, 2, 3… 18 on side 1, track 0.

This disparity means that side 0 will read compatibly across all drives, but side 1 will have scrambled sector order when read and written between drives using different side 1 mappings. This will be reflected as file system corruption in DOS.

The Black Box Floppy Board has the unique ability to select between XF551, Percom, and ATR8000 compatible side 1 mappings by a setting in its configuration screen.
