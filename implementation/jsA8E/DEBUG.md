# Debug

> Hardware emulation reference: Before implementing any Atari 800 XL PAL machine related hardware emulation, use the [AHRM](/AHRM/index.md) as reference.

- Files: `jsA8E/js/app/ui.js`, `jsA8E/js/core/debugger.js`, `jsA8E/js/core/atari.js`, `jsA8E/js/core/app_proxy.js`, `jsA8E/emulator_worker.js`, `jsA8E/js/render/gl.js`, `jsA8E/js/core/cpu.js`, `jsA8E/js/core/hostfs.js`, `jsA8E/index.html`
- Purpose: document browser (`jsA8E/`) debug/diagnostic hooks, error-reporting paths, and current runtime limits.
- Status: verified on 2026-03-09 (`implemented`, with usability limits below).
- Notes: an optional CPU register overlay path exists: `ui.js` reads `debugEl` from `#debug`, and `atari.js` writes `PC/A/X/Y/SP/P` via `updateDebug()` (per painted frame, reset, and initial paint). In the shipped UI, `index.html` has no `#debug` element, so this overlay is inactive by default. Runtime debug state and stepping/breakpoint orchestration are split into `js/core/debugger.js`, while `atari.js` now delegates through the same public app APIs (`setBreakpoints`, `stepInstruction`, `stepOver`, `getDebugState`, `onDebugStateChange`) in both main-thread and worker mode (`app_proxy.js` + `emulator_worker.js`). Worker mode forwards throttled frame debug snapshots and immediate breakpoint/pause/step updates to the UI channel. CPU PC hooks (`setPcHook` / `clearPcHook` in `cpu.js`) now support debugger-controlled early run termination (`ctx.breakRun`) and instruction counting (`instructionCounter`) so assembler breakpoints and stepping can stop execution deterministically without tight-looping. The CPU run loop also emits per-instruction trace samples into `debugger.js`, which maintains a small rolling `(pc,a,x,y,sp,p,cycles)` trace buffer for automation and post-mortem artifact capture. Unsupported opcodes now surface as deterministic `fault_illegal_opcode` stops, and runtime execution exceptions are normalized as `fault_execution_error` snapshots instead of silently disappearing inside the automation path. Debug snapshots export both `cycleCounter` and `instructionCounter`, and the debugger runtime exposes paused-mode bounded execution via `runUntilPc(targetPc, options)` with step-over built on the same primitive instead of routing through the frame loop. `memory.js` now provides `readMemory`, `readRange`, and `getBankState`; `atari.js` packages those together with `captureScreenshot()` and `collectArtifacts()`. `window.A8EAutomation` builds on that with `debug.getSourceContext(...)`, `debug.disassemble(...)`, wait/event helpers, HostFS-backed assembly, and post-fault inspection flows. Screenshot capture is framebuffer-based (`blitViewportToImageData`) so output is stable across software/WebGL/worker rendering paths and independent of browser canvas scaling. Worker diagnostics remain active: worker-side failures are wrapped as `{ type: "error", message }`, forwarded to main-thread proxy, and logged via `console.error` (`app_proxy.js`), with `worker.onerror` as an additional failure path. Rendering diagnostics are exception-driven: `gl.js` throws detailed shader/program/framebuffer errors; both main-thread and worker init paths catch WebGL setup failures and fall back to 2D canvas. Additional console diagnostics include keyboard CSS scaling mismatch warning (once), file/auto-load errors, and H: host filesystem listener/IndexedDB init errors.
- Issues: there is still no dedicated standalone diagnostics panel in the shipped UI; runtime/source disassembly is currently exposed through the browser automation API rather than a visible global debug panel.
- Todo: add a visible debug panel toggle (`#debug`) and optional in-UI trace/disassembly view; expose worker/render backend and last error in UI; add optional runtime log categories to reduce reliance on ad-hoc console output; if trace consumers need more context later, prefer extending the bounded trace record shape over increasing the ring size aggressively.
