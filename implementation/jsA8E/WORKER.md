# Worker Boundary

- Files: `jsA8E/emulator_worker.js`, `jsA8E/js/core/app_proxy.js`
- Purpose: run emulation away from the main thread and exchange events/data with UI/audio.
- Status: verified on 2026-03-11 (`implemented`).
- Notes: worker messaging transports control/state, hostfs snapshots, and audio bridge status between runtime parts; rendering is driven through OffscreenCanvas with fallback to legacy in-thread mode when unsupported. The proxy/worker boundary now supports request/response RPC in addition to fire-and-forget control messages so external automation can fetch structured results from worker mode without special cases. Debugger protocol includes runtime debug snapshots (`debugState`) and control commands for assembler debugging (`setBreakpoints`, `stepInstruction`, `stepOver`) alongside regular run/pause/reset flow, plus automation-oriented calls for `getDebugState`, `getCounters`, `getTraceTail`, `runUntilPc`, `readMemory`, `readRange`, `getBankState`, `getMountedDiskForDeviceSlot`, `getConsoleKeyState`, `captureScreenshot`, `collectArtifacts`, `saveSnapshot`, and `loadSnapshot`. Worker lifecycle requests (`start`, `pause`, `reset`) now use the same RPC path as debugger/media requests, so automation waits for explicit worker acknowledgements instead of racing the next debug-state read. Pending RPC calls now carry per-command wall-clock timeouts and reject with structured command-aware errors when the worker misses a response. XEX and disk mounting use the detailed request/response path (`loadDiskToDeviceSlot` as RPC) so worker-backed automation receives structured preflight results and serialized loader error details instead of losing them in fire-and-forget command logging. Snapshot RPC now also transfers the binary snapshot buffer directly, while `loadSnapshot` responses reuse the normal worker control-ack state payload so proxy-side run/ROM/media/debug status stays in sync after a restore. Worker state snapshots now include a `config` object (turbo, sioTurbo, audioEnabled, optionOnStart, keyboardMappingMode) and the proxy tracks these fields, exposing matching getters so the UI can reconcile toggle button states after snapshot restore or other internal config changes. The proxy setters also update local state eagerly so getters stay correct between worker state broadcasts. Reset commands still forward optional cold-reset overrides such as initial `portB`, and the browser boot path now supports a supported no-worker fallback (`?a8e_worker=0` or `window.A8E_BOOT_OPTIONS.worker = false`) for harness/debug runs where the worker backend itself is the suspect component. Worker automation still has an explicit on-demand debug-state request path, and worker debug snapshots preserve pause/fault metadata (`stopAddress`, `faultType`, `faultMessage`, `faultAddress`, `opcode`) so `window.A8EAutomation` sees the same stop semantics in worker mode as in the main-thread runtime. Binary request/response values are normalized across realms so iframe/worker automation can pass foreign `ArrayBuffer` views, `DataView`, and copied screenshot/artifact buffers without realm-identity failures, and falsy payloads such as `0` or `false` survive the round-trip.
- Issues: browser-driven Chrome headless runs still need more direct verification with the new no-worker fallback before this area can be called fully signed off for harness use.
- Todo: log protocol changes whenever message schema changes; keep new RPC result payloads backward-compatible when adding worker methods because `window.A8EAutomation` depends on them in both worker and main-thread mode.
