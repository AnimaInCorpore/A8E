# Core Emulation

- Files: `jsA8E/js/core/cpu.js`, `jsA8E/js/core/cpu_tables.js`, `jsA8E/js/core/antic.js`, `jsA8E/js/core/gtia.js`, `jsA8E/js/core/pokey.js`, `jsA8E/js/core/pokey_sio.js`, `jsA8E/js/core/memory.js`, `jsA8E/js/core/io.js`, `jsA8E/js/core/atari.js`, `jsA8E/js/core/hw.js`, `jsA8E/js/core/playfield.js`, `jsA8E/js/core/playfield_renderer.js`, `jsA8E/js/core/state.js`, `jsA8E/js/core/snapshot_codec.js`
- Purpose: mirror Atari hardware behavior in JavaScript with timing-compatible execution.
- Status: updated on 2026-03-11 (`partial`).
- Notes: CPU/ANTIC/GTIA/POKEY flow is coordinated in core modules with shared machine state. `playfield.js` now owns playfield line orchestration, scratch-buffer setup, and active-line geometry, while `playfield_renderer.js` owns the per-mode pixel generators for ANTIC modes 2-F plus shared blank-line/background draw helpers. The renderer path still follows the legacy-style active-line geometry, HSCROL timing, visible PMG interleave, and blank-line color-burst behavior used by the native core. Character-mode vertical scrolling still clamps out-of-range glyph rows to blank data instead of reading neighboring character bytes, which avoids stray horizontal artifact lines on scrolled text/bitmap hybrid screens. Modes 4 and 5 still handle the character byte's bit 7 directly as the alternate-color-set selector instead of routing through the Mode 2/3 `decodeTextModeCharacter` helper, matching the native core and real hardware behavior. The JS path still renders scrolled lines through a wider scratch buffer and copies the visible 456-pixel window back into the main video buffer. XEX mount preflight now simulates the file's writes in load order, so a segment that writes `D301` can bank ROM out before later bytes target `$5000-$57FF`, `$A000-$BFFF`, `$C000-$CFFF`, or `$D800-$FFFF`; only writes that are still ROM-backed at the time they execute are rejected. When no explicit reset-time `PORTB` override is supplied, that preflight now also inherits the current `Option-on-Start` policy, so BASIC-area XEX content is checked against the same BASIC-off boot expectation that the OS boot shim will use. The browser core now also derives the default reset `PORTB` value from the real `IO_PORTB` initialization entry (`$FD`) instead of indexing the init-value list by address, which keeps XEX preflight and reset-time banking aligned with the XL memory map. Event scheduling now stays on `cycleCounter` for DLI, POKEY timers, and SIO so the current instruction-granular CPU model does not drift against the newer draw-path timing work over longer runs. The core runtime now also supports paused-state full snapshot save/load through a versioned binary container (`snapshot_codec.js`). Snapshot payloads capture CPU registers/counters, RAM + shadow RAM, video buffers, `ioData` timing/custom-chip state, mounted media/ROM bytes, debugger trace/breakpoints, input bookkeeping, and the H: device's persisted HostFS file set plus open-channel state so automation and UI quick-save flows can round-trip a self-contained machine image.
