# Core Emulation

> Hardware emulation reference: Before implementing any Atari 800 XL PAL machine related hardware emulation, use the [AHRM](/AHRM/index.md) as reference.

- Files: `jsA8E/js/core/cpu.js`, `jsA8E/js/core/cpu_tables.js`, `jsA8E/js/core/antic.js`, `jsA8E/js/core/gtia.js`, `jsA8E/js/core/pokey.js`, `jsA8E/js/core/pokey_sio.js`, `jsA8E/js/core/memory.js`, `jsA8E/js/core/io.js`, `jsA8E/js/core/atari.js`, `jsA8E/js/core/hw.js`, `jsA8E/js/core/playfield.js`, `jsA8E/js/core/playfield_renderer.js`, `jsA8E/js/core/state.js`, `jsA8E/js/core/snapshot_codec.js`
- Purpose: mirror Atari hardware behavior in JavaScript with timing-compatible execution.
- Status: updated on 2026-04-03 (`partial`).
- Notes: CPU/ANTIC/GTIA/POKEY flow is coordinated in core modules with shared machine state. `playfield.js` now owns playfield line orchestration, scratch-buffer setup, and active-line geometry, while `playfield_renderer.js` owns the per-mode pixel generators for ANTIC modes 2-F plus shared blank-line/background draw helpers. The renderer path still follows the legacy-style active-line geometry, HSCROL timing, visible PMG interleave, and blank-line color-burst behavior used by the native core. Active-line geometry now anchors the unscrolled playfield window to the corrected AHRM 4.14 width starts (clock 26/18/10 after HSYNC for narrow/normal/wide), and HSCROL-promoted windows delay from those baselines by one cycle per two color clocks of scroll. Scrolled non-wide lines now preserve already-drawn PMG pixels in both the clipped aperture fill and the trailing border background fill, so the HSCROL compensation path no longer punches holes through ladders or other player/missile graphics. Character-mode vertical scrolling still clamps out-of-range glyph rows to blank data instead of reading neighboring character bytes, which avoids stray horizontal artifact lines on scrolled text/bitmap hybrid screens. Modes 4 and 5 still handle the character byte's bit 7 directly as the alternate-color-set selector instead of routing through the Mode 2/3 `decodeTextModeCharacter` helper, matching the native core and real hardware behavior. **Mode 3 now correctly reads CHBASE (character set base) once at scanline start instead of per-character**, eliminating mid-frame charset glitches when DLI changes CHBASE during rendering. All text modes (2-3) now align on baseline behavior: each scanline's characters render with consistent charset throughout the line, and CHBASE changes from DLI take effect at the next scanline. PMG DMA fetches now treat `VDELAY` as an odd-scanline load mask instead of shifting PMG memory rows, so delayed sprites hold the previous latch on even scan lines, and player DMA keeps the missile slot active as required by AHRM 4.13. The JS path still renders scrolled lines through a wider scratch buffer and copies the visible 456-pixel window back into the main video buffer. XEX mount preflight now simulates the file's writes in load order, so a segment that writes `D301` can bank ROM out before later bytes target `$5000-$57FF`, `$A000-$BFFF`, `$C000-$CFFF`, or `$D800-$FFFF`; only writes that are still ROM-backed at the time they execute are rejected. When no explicit reset-time `PORTB` override is supplied, that preflight now also inherits the current `Option-on-Start` policy, so BASIC-area XEX content is checked against the same BASIC-off boot expectation that the OS boot shim will use. The browser core now also derives the default reset `PORTB` value from the real `IO_PORTB` initialization entry (`$FD`) instead of indexing the init-value list by address, which keeps XEX preflight and reset-time banking aligned with the XL memory map. DLI deadlines follow the raster beam clock (`io.clock`) while timer/SIO deadlines stay on `cycleCounter`, and the scheduler tracks separate beam/master wake thresholds so scanline rendering and CPU execution each poll the right clock domain. Per AHRM 4.8, ANTIC asserts NMI at cycle 8 on the last scanline of a DLI-enabled mode line; the NMI handler entry sequence begins at the first instruction boundary on cycle 10 or later. Same-scanline `NMIEN` writes now follow the AHRM DLI gate: writes by cycle 7 enable the current-line DLI, writes on cycle 8 can still suppress it, and an exact cycle-7 enable postpones the CPU-visible DLI by one beam cycle while leaving NMIST behavior unchanged. VBI fires at the beginning of scan line 248 (AHRM 4.8). VCOUNT now reflects the next scanline starting on cycle 111 of the previous line per AHRM 4.10, including the one-cycle PAL end-of-frame anomaly (`$9C` on the final scanline's cycle 111 before wrapping to `$00`). ANTIC JVB decode now ignores DLI/vertical-scroll bits (`$41/$C1/...`), JVB+DLI replay re-arms one DLI per scanline until vertical blank, and NMIST DLI bit (bit 7 per AHRM 4.8) lifetime matches AHRM (cleared at scan line 248 or `NMIRES`, not every scanline). The core runtime now also supports paused-state full snapshot save/load through a versioned binary container (`snapshot_codec.js`). Snapshot saves now default to advancing paused execution to the next frame boundary before encoding, which avoids restoring from unstable mid-frame raster state; callers can still opt out with `timing: "exact"`. Snapshot payloads capture CPU registers/counters, RAM + shadow RAM, video buffers, `ioData` timing/custom-chip state, mounted media/ROM bytes, debugger trace/breakpoints, input bookkeeping, and the H: device's persisted HostFS file set plus open-channel state so automation and UI quick-save flows can round-trip a self-contained machine image. ANTIC display-list command bits are now named constants (`ANTIC_DLI_BIT`, `ANTIC_LMS_BIT`, `ANTIC_MODE_BITS`, etc.) for improved code clarity and AHRM reference alignment. WSYNC handler uses `WSYNC_CYCLE` = 105 and `WSYNC_BOUNDARY` = 104 matching AHRM 4.9: writes up to cycle 103 wait on the current line, writes at cycle 104 or later wait for the next line. DMA steal positions match AHRM 4.14: refresh at cycles 25/29/33/37/41/45/49/53/57 (one-cycle defer, further blocked drops), display list instruction at cycle 1, LMS/jump address at cycles 6–7, missile at cycle 0, players at cycles 2–5 (AHRM 4.13). Playfield DMA no longer gets charged as a bulk line-start stall. `renderer_base.js` now schedules per-cycle playfield steals, buffers first-row display/name fetches into a 48-byte line buffer, and switches late fetches after cycle 105 to the virtual CPU bus / refresh-drop path instead of stalling the CPU. Character modes 2-7 now separate the display/name fetch from the later `+3` cycle character-data fetch slot, while modes 8-F reuse buffered display bytes on repeated scanlines. `playfield_dma_contention_regression.test.js` now pins the cycle-105 scheduling boundary, cycle-106 virtual fetch behavior, and refresh-drop line-buffer artifact reuse. The GTIA priority buffer is 16-bit (`Uint16Array`), and all four player/missile draw functions (`drawPlayer`, `drawMissile`, `drawPlayerSpan`, `drawMissileSpan`) mask existing priority with `& 0xffff`. Priority constants `PRIO_PF3` and `PRIO_M10_PM0-3` are wired through the full config chain (`atari.js → antic.js → A8EPlayfield → A8EPlayfieldRenderer → renderer_base → mode_*.js`). Inline missile priority mask blocks in `drawPlayerMissiles` are delegated to the shared `missilePriorityMask()` helper, and all four player inline masks include the appropriate `PRIO_M10_PM*` bit. `clockAction()` in `renderer_base.js` updates VCOUNT before DMA handling, matching the C core ordering. Undocumented ANE/LXA handling now mirrors the native core's fake6502-compatible magic constants, keeping the JS and C CPU cores aligned with the Lorenz opcode suite. The browser CPU now also mirrors the native core for the rest of the fake6502 Lorenz set, including `ARR`/`LAS`/`SHA`/`SHX`/`SHY`/`TAS`/`RRA`/`SBX`, plus the `SHX`/`SHY` store-address glitch and the `RRA`/`ISC` decimal-cycle cancellation.
