# Core Emulation

- Files: `jsA8E/js/core/cpu.js`, `jsA8E/js/core/cpu_tables.js`, `jsA8E/js/core/antic.js`, `jsA8E/js/core/gtia.js`, `jsA8E/js/core/pokey.js`, `jsA8E/js/core/pokey_sio.js`, `jsA8E/js/core/memory.js`, `jsA8E/js/core/io.js`, `jsA8E/js/core/atari.js`, `jsA8E/js/core/hw.js`, `jsA8E/js/core/playfield.js`, `jsA8E/js/core/playfield_renderer.js`, `jsA8E/js/core/state.js`
- Purpose: mirror Atari hardware behavior in JavaScript with timing-compatible execution.
- Status: updated on 2026-03-11 (`partial`).
- Notes: CPU/ANTIC/GTIA/POKEY flow is coordinated in core modules with shared machine state. `playfield.js` now owns playfield line orchestration, scratch-buffer setup, and active-line geometry, while `playfield_renderer.js` owns the per-mode pixel generators for ANTIC modes 2-F plus shared blank-line/background draw helpers. The renderer path still follows the legacy-style active-line geometry, HSCROL timing, visible PMG interleave, and blank-line color-burst behavior used by the native core. Character-mode vertical scrolling still clamps out-of-range glyph rows to blank data instead of reading neighboring character bytes, which avoids stray horizontal artifact lines on scrolled text/bitmap hybrid screens. Modes 4 and 5 still handle the character byte's bit 7 directly as the alternate-color-set selector instead of routing through the Mode 2/3 `decodeTextModeCharacter` helper, matching the native core and real hardware behavior. The JS path still renders scrolled lines through a wider scratch buffer and copies the visible 456-pixel window back into the main video buffer. Event scheduling now stays on `cycleCounter` for DLI, POKEY timers, and SIO so the current instruction-granular CPU model does not drift against the newer draw-path timing work over longer runs.
