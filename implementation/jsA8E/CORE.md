# Core Emulation

> Hardware emulation reference: Before implementing any Atari 800 XL PAL machine related hardware emulation, use the [AHRM](/AHRM/index.md) as reference.

- Files: `jsA8E/js/core/cpu.js`, `jsA8E/js/core/cpu_tables.js`, `jsA8E/js/core/antic.js`, `jsA8E/js/core/gtia.js`, `jsA8E/js/core/pokey.js`, `jsA8E/js/core/pokey_sio.js`, `jsA8E/js/core/memory.js`, `jsA8E/js/core/io.js`, `jsA8E/js/core/atari.js`, `jsA8E/js/core/hw.js`, `jsA8E/js/core/playfield.js`, `jsA8E/js/core/playfield_renderer.js`, `jsA8E/js/core/state.js`, `jsA8E/js/core/snapshot_codec.js`
- Purpose: mirror Atari hardware behavior in JavaScript with timing-compatible execution.
- Status: updated on 2026-03-15 (`partial`).
- Notes: CPU/ANTIC/GTIA/POKEY flow is coordinated in core modules with shared machine state. `playfield.js` now owns playfield line orchestration, scratch-buffer setup, and active-line geometry, while `playfield_renderer.js` owns the per-mode pixel generators for ANTIC modes 2-F plus shared blank-line/background draw helpers. The renderer path still follows the legacy-style active-line geometry, HSCROL timing, visible PMG interleave, and blank-line color-burst behavior used by the native core. Character-mode vertical scrolling still clamps out-of-range glyph rows to blank data instead of reading neighboring character bytes, which avoids stray horizontal artifact lines on scrolled text/bitmap hybrid screens. Modes 4 and 5 still handle the character byte's bit 7 directly as the alternate-color-set selector instead of routing through the Mode 2/3 `decodeTextModeCharacter` helper, matching the native core and real hardware behavior. **Mode 3 now correctly reads CHBASE (character set base) once at scanline start instead of per-character**, eliminating mid-frame charset glitches when DLI changes CHBASE during rendering. All text modes (2-3) now align on baseline behavior: each scanline's characters render with consistent charset throughout the line, and CHBASE changes from DLI take effect at the next scanline. The JS path still renders scrolled lines through a wider scratch buffer and copies the visible 456-pixel window back into the main video buffer. XEX mount preflight now simulates the file's writes in load order, so a segment that writes `D301` can bank ROM out before later bytes target `$5000-$57FF`, `$A000-$BFFF`, `$C000-$CFFF`, or `$D800-$FFFF`; only writes that are still ROM-backed at the time they execute are rejected. When no explicit reset-time `PORTB` override is supplied, that preflight now also inherits the current `Option-on-Start` policy, so BASIC-area XEX content is checked against the same BASIC-off boot expectation that the OS boot shim will use. The browser core now also derives the default reset `PORTB` value from the real `IO_PORTB` initialization entry (`$FD`) instead of indexing the init-value list by address, which keeps XEX preflight and reset-time banking aligned with the XL memory map. DLI deadlines follow the raster beam clock (`io.clock`) while timer/SIO deadlines stay on `cycleCounter`, and the scheduler tracks separate beam/master wake thresholds so scanline rendering and CPU execution each poll the right clock domain. Per AHRM 4.8, ANTIC asserts NMI at cycle 8 on the last scanline of a DLI-enabled mode line; the NMI handler entry sequence begins at the first instruction boundary on cycle 10 or later. VBI fires at the beginning of scan line 248 (AHRM 4.8). VCOUNT reads reflect the next-scanline value from cycle 100 onward (AHRM 4.14 event timing chart). ANTIC JVB decode now ignores DLI/vertical-scroll bits (`$41/$C1/...`), JVB+DLI replay re-arms one DLI per scanline until vertical blank, and NMIST DLI bit (bit 7 per AHRM 4.8) lifetime matches AHRM (cleared at scan line 248 or `NMIRES`, not every scanline). The core runtime now also supports paused-state full snapshot save/load through a versioned binary container (`snapshot_codec.js`). Snapshot saves now default to advancing paused execution to the next frame boundary before encoding, which avoids restoring from unstable mid-frame raster state; callers can still opt out with `timing: "exact"`. Snapshot payloads capture CPU registers/counters, RAM + shadow RAM, video buffers, `ioData` timing/custom-chip state, mounted media/ROM bytes, debugger trace/breakpoints, input bookkeeping, and the H: device's persisted HostFS file set plus open-channel state so automation and UI quick-save flows can round-trip a self-contained machine image. ANTIC display-list command bits are now named constants (`ANTIC_DLI_BIT`, `ANTIC_LMS_BIT`, `ANTIC_MODE_BITS`, etc.) for improved code clarity and AHRM reference alignment. WSYNC handler uses `WSYNC_CYCLE` = 105 and `WSYNC_BOUNDARY` = 104 matching AHRM 4.9: writes up to cycle 103 wait on the current line, writes at cycle 104 or later wait for the next line. DMA steal positions match AHRM 4.14: refresh at cycles 25/29/33/37/41/45/49/53/57 (one-cycle defer, further blocked drops), display list instruction at cycle 1, LMS/jump address at cycles 6–7, missile at cycle 0, players at cycles 2–5 (AHRM 4.13). Playfield DMA cycle consumption remains direct (`ctx.cycleCounter++`) rather than going through the event-aware `stealDma()` path, which allows rendering to advance cycles at its own pace without triggering interim timed events.
- Issues: broader regression verification against real raster-effect content is still open. Known AHRM discrepancies: (1) `DLI_HORIZONTAL_OFFSET` is 14, not 8: AHRM 4.8 says ANTIC asserts NMI at cycle 8, but the emulator schedules the DLI event at cycle 14 from line start, approximately combining the NMI pin assertion (cycle 8), the 2-cycle CPU acknowledgment, and partial instruction-finish time into one offset. (2) VBI fires on scan line `LAST_VISIBLE_LINE + 2` (249) instead of 248: AHRM 4.8 says VBI is asserted at the beginning of scan line 248; the code clears the DLI status bit on line 248 (correct) but delays the VBI NMI assertion to line 249 (one line late).
